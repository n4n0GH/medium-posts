Research & Development — Your Money At Work

Let me give you an update on the current happenings around 0xchan.

First of all we started moving our own infrastructure in place to host an instance of 0xchan; this included, but is not limited to, setting up a proper VPS and IPFS node. The structure for the contracts is also finalized and ready to be turned from pseudo- into actual code. So that’s a good start.

Main point I wanted to address is the development of the frontend; as some of you may know, I recently developed an idle game (found at https://github.com/n4n0GH/curvy-idle-game), that has absolutely nothing to do with 0xchan. What’s up with that? To explain this, I need to go back a few steps and explain what we’re planning with the stand-alone app.

In the beginning of our research phase we thought about using one of the many available chan sources and build 0xchan on that base, but after some trial and error and encountering heaps of undocumented spaghetticode we decided against using existing systems, not only for our own sake, but for the sake of publishing a well documented end product ourselves.

So we had to find a viable solution; one that would allow for rapid prototyping but was also mature enough to be used for 0xchan. Choices right now are really only between few frameworks like React, VueJS or Angular. Research made us choose VueJS over the others because we found a lot of good documentation from other teams who built desktop apps with the use of Electron and VueJS in combination with IPFS that enable us to avoid making the same mistakes those teams did.

With VueJS set as the underlying framework to facilitate this venture, I had to do some extensive learning as I was familiar with the general concept of it, but not it’s detailed features and pit-falls. So I studied it with paid and free material while simultaneously prototyping components for 0xchan to use.

Once I was finished with my paid study material, I went on to set my learned things into practice and made the idle game I mentioned earlier. This allowed me to not only work on a streamlined code structure, but also to test some things during development that will come in handy for the inevitable 0xchan rewrite.

Right now 0xchan exists in a mess of code. But it’s sort of working (with offline data). Once all parts have been assembled I’ll work on refactoring everything, making components that make sense and get it to a state of minimal code, well documented and up on Github for you to look at or clone. In fact, this will be a base for anyone looking to make a new chan with whatever Database they decide to use, but a more modern framework underneath, so I guess that’s a nice thing to have.

How does this work in conjunction with the desktop application though?

One possibility to create a desktop application that makes it truly impossible to stop the use of 0xchan is to let users run a client on their machine that hosts a local webserver and IPFS node (optional). Everything gets served to the user’s browser and they can user 0xchan like they would any other website. We favored this approach over another idea we had where the user would essentially run a sandboxed browser environment with it’s own metamask implemented. The security risk for this is too high when users run outdated versions of the client and some issue was found in the metamask implementation so we’d rather not have you risk losing your Ether due to that.

The chosen approach could potentially also run on a smartphone or tablet that supports running those processes (mainly node.js) on their machines and then users could access it via Trustwallet or other applications that allow to interact with Web3 components.

So how far are we away from an actual first release? I can see a testnet release with a fully rewritten codebase to try out functionality with a broader userbase happen around the end of September to mid October if development can proceed like it did so far (i.e. no one ends up in hospital or something like that).

We’re all gonna make it.
